# 10 — Spreadsheet Integration (Excel + Google Sheets)

## Why this exists

The JD calls out two skills directly:

> "Advanced Excel/Google Sheets skills, including financial modeling and data analysis"
> "Familiarity with revenue recognition principles (ASC 606)"

A working multi-tab spreadsheet workbook with live formulas demonstrates both — at the same time, on the same artifact. The HM opens the workbook, edits the discount cell, and watches the ASC 606 revenue recognition schedule recalculate across all 36 contract months. That's harder to fake than any resume bullet.

This is also the highest-leverage deliverable: spreadsheets travel inside organizations. The HM forwards it to a colleague. The colleague forwards it to their boss. Every forward is the artifact selling itself.

## Two-tier integration

### Tier 1: Always-on `.xlsx` export — *(Phase 7)*

Every deal review generates a downloadable `.xlsx` workbook with full formulas, conditional formatting, and charts. Works in Excel, Numbers, LibreOffice, and Google Sheets (via upload). **No authentication required.** Becomes the 5th artifact in the deal review's artifacts panel, alongside the redlined MSA, order form, AE email, and approval one-pager.

This tier is non-negotiable. Every demo run produces this.

### Tier 2: Live Google Sheets integration — *(Phase 8, optional)*

A "Open as live Google Sheets model" button on the deal review page. Server-side uses the Google Drive API to copy a master template Sheet, populate it with this deal's values, and return a public shareable link. The visitor clicks → opens in Sheets → uses `File > Make a copy` to get their own editable workbook.

This tier is the optional bonus. Strong if it lands, but the .xlsx alone is enough to deliver the JD signal.

## The workbook structure (10 tabs)

The same structure is used for both the .xlsx export and the Google Sheets template. Build the .xlsx generator first; the Sheets template is essentially a copy of the same workbook hosted in Drive.

### Tab 1 — Deal Summary

Static metadata, formatted as a clean header block.

| Cell | Value |
|---|---|
| Customer | {customer.name} |
| Customer segment | {customer.segment} |
| Industry | {customer.industry} |
| Deal type | {deal.deal_type} |
| Deal stage | {deal.stage} |
| AE owner | {deal.ae_owner} |
| ACV (proposed) | {deal.acv} |
| TCV (proposed) | {deal.tcv} |
| Term length (months) | {deal.term_months} |
| Discount % (headline) | {deal.discount_pct} |
| Pricing model | {deal.pricing_model} |
| Generated at | {now} |
| Generated by | Kiln ([DEMO_URL]) |

Cell styling: header row in clay accent (#C2410C), 14pt Inter Bold; body in 11pt Inter Regular. Number cells use accounting format with $ prefix.

### Tab 2 — Pricing Model (LIVE FORMULAS)

This is the core of the workbook. **Inputs go in yellow-highlighted cells; formulas go in white cells.** The visitor edits inputs and watches outputs recalculate.

**Inputs (yellow):**
- `B3`: List price (annual, USD)
- `B4`: Discount % (decimal, e.g., 0.15 for 15%)
- `B5`: Term length (months)
- `B6`: Ramp length (months) — months at reduced pricing
- `B7`: Ramp price multiplier (decimal, e.g., 0.5 for 50%)
- `B8`: Free month count
- `B9`: Annual price escalator (decimal, default 0)
- `B10`: Assumed gross margin at list (decimal, default 0.40)

**Calculated outputs (white):**
- `D3`: Discounted price = `=B3*(1-B4)`
- `D4`: Effective discount % over term = formula accounting for ramp + free months
- `D5`: Year 1 ACV = formula
- `D6`: Year 2 ACV = formula (with escalator if applicable)
- `D7`: Year 3 ACV = formula
- `D8`: TCV = `=SUM(D5:D7)` (or however many years)
- `D9`: Estimated margin $ = `=D8*B10` (with discount-adjusted variant in D10)
- `D10`: Estimated margin % = formula
- `D11`: Margin haircut from discount = formula

**Conditional formatting:**
- D4 (effective discount) turns red if > 30%, amber if > 20%, green if ≤ 20%
- D10 (margin %) turns red if < 25%, amber if 25–30%, green if ≥ 30%

**Disclaimer cell (B14):** "Margin calculations assume {gross_margin}% gross margin at list. Real margin depends on COGS not visible to this model. Treat as directional."

### Tab 3 — ASC 606 Revenue Recognition Schedule (LIVE FORMULAS)

A monthly schedule for the full term. If term = 36 months, this tab has 36 data rows.

**Columns:**
| Col | Header | Formula |
|---|---|---|
| A | Month # | `1`, `2`, ..., `36` |
| B | Calendar month | `=DATE(YEAR(start_date), MONTH(start_date)+A-1, 1)` |
| C | Performance obligation phase | `="Ramp"` if `A<=ramp_length`, `="Free"` if free month, else `="Standard"` |
| D | Subscription revenue recognized | formula referencing Pricing Model tab |
| E | Usage revenue recognized (estimated) | formula based on usage commit |
| F | Variable consideration adjustment | formula for rollover/true-up |
| G | Total recognized this month | `=D+E+F` |
| H | Cumulative recognized | `=G + previous_row_H` |
| I | Deferred revenue balance | `=cumulative_billed - H` |
| J | Notes | static, e.g., "Ramp month — 50% billing" |

**Bottom rows:**
- Row 38: Subtotals = `=SUM()` for each numeric column
- Row 40: Validation check: `=IF(SUM(G:G)=Pricing!D8, "✓ Reconciles to TCV", "✗ Mismatch")` — green/red conditional formatting

**Right side of the tab:**
A small bar chart showing recognized revenue by quarter for the term. Auto-generated from the data.

This tab is the showcase. It's a real ASC 606 model with real formulas, not a screenshot of one.

### Tab 4 — Alternative Structures (side-by-side comparison)

Four columns: `As Proposed | Alt 1 | Alt 2 | Alt 3`. Each column has the same input set as the Pricing Model tab, with different values populated by Kiln's Pricing Agent's `alternative_structures` output.

Below the inputs, calculated outputs:
- TCV
- Effective discount %
- Y1 ACV
- Margin %
- Margin $
- ASC 606 recognition pattern (text)

**Winner highlighting:** the highest TCV value across the 4 columns gets a green background. Lowest margin gets red. Helps the visitor spot which alternative wins on which dimension.

A short "Recommendation" section at the bottom pulls from the agent's reasoning_summary.

### Tab 5 — Approval Routing

Reads the Approval Matrix tab (Tab 6) and applies its rules to the deal data on Tab 1.

**Output:**
- Required approvers list (driven by formulas like `=IF(deal.acv>500000, "CFO", "")`)
- Expected approval cycle time (computed from approver set)
- Routing rationale (text, generated by Kiln's Approval Agent)

This tab demonstrates that the approval logic is **transparent and editable** — the HM can see exactly which rule fired and why.

### Tab 6 — Approval Matrix (EDITABLE)

The full approval matrix as a table. Each row is a rule:

| Rule Name | Condition | Required Approver | Priority | Active |
|---|---|---|---|---|
| High discount approval | `discount_pct > 0.25` | rev_ops | 1 | TRUE |
| Large ACV CFO approval | `acv > 500000` | cfo | 2 | TRUE |
| MFN clause CEO approval | `clauses includes "MFN"` | ceo | 1 | TRUE |
| ... | ... | ... | ... | ... |

The conditions use Excel/Sheets-friendly logical operators. **Editing a rule changes the routing on Tab 5** via formulas.

This is a power-user feature. Most HMs won't edit it, but the ones who do will be impressed that it works.

### Tab 7 — Pricing Guardrails

Read-only table of pricing guardrails with pass/fail status:

| Guardrail | Threshold | Actual | Status | Severity |
|---|---|---|---|---|
| Max discount Enterprise | 30% | {actual} | PASS/FAIL | Block w/ approval |
| Min margin floor | 25% | {actual} | PASS/FAIL | Warn |
| Max ramp length | 6mo | {actual} | PASS/FAIL | Warn |
| ... | ... | ... | ... | ... |

Status cell uses conditional formatting (green for PASS, red for FAIL).

### Tab 8 — Similar Past Deals

The top-3 vector search results, plus 5 additional historical deals for broader context. Each row:

| Customer | Deal Type | ACV | Discount % | Term | Outcome | Decision Note |
|---|---|---|---|---|---|---|

Below the table:
- A small bar chart: discount % across similar deals (highlights this deal's discount in red)
- A scatter plot: ACV vs. discount %, with this deal as a labeled point

### Tab 9 — Comp Analysis (data analysis showcase)

A pivot-table-style analysis of the broader deal pipeline. Shows:

- Average discount % by customer segment (Enterprise / Mid-market / PLG)
- Average ACV by deal type (New / Expansion / Renewal / Partnership)
- Win rate by AE (using Tab 8 data)
- Median deal cycle time by approval depth

Includes 2–3 charts auto-generated from the data.

This tab exists to signal **data analysis fluency**. The HM scans it and sees: this person can do pivot-table-style aggregation in Sheets.

### Tab 10 — Audit Log

Static dump of the audit log entries for this review. Read-only. Each row is one agent decision with timestamp, agent name, duration, tokens used, and a "view reasoning" link back to the live Kiln deal page.

## Implementation

### Tier 1: .xlsx generation (Phase 7)

**Library**: `exceljs` (mature, supports formulas, conditional formatting, charts).

```bash
npm install exceljs
```

**File location**: `lib/document-templates/spreadsheet.ts`

**Generator function signature**:

```ts
export async function generateDealSpreadsheet(review: DealReview): Promise<Buffer> {
  const workbook = new ExcelJS.Workbook();
  workbook.creator = "Kiln";
  workbook.created = new Date();
  
  buildDealSummaryTab(workbook, review);
  buildPricingModelTab(workbook, review);
  buildAsc606ScheduleTab(workbook, review);
  buildAlternativeStructuresTab(workbook, review);
  buildApprovalRoutingTab(workbook, review);
  buildApprovalMatrixTab(workbook, review);
  buildPricingGuardrailsTab(workbook, review);
  buildSimilarDealsTab(workbook, review);
  buildCompAnalysisTab(workbook, review);
  buildAuditLogTab(workbook, review);
  
  return await workbook.xlsx.writeBuffer();
}
```

Each `build*Tab` function takes the workbook and the review object and adds the structured tab. Keep these functions in separate files under `lib/document-templates/spreadsheet/<tab>.ts` for readability.

**Critical implementation details:**

1. **Use real formulas, not pre-computed values.** ExcelJS supports `cell.value = { formula: 'B3*(1-B4)' }`. The whole point is that when the visitor edits a cell, the spreadsheet recalculates.

2. **Reference cells across tabs.** Pricing Model formulas should reference `Tab1!B7` (the Discount % cell). When the visitor changes a cell on the Pricing Model tab, the ASC 606 schedule on Tab 3 must recalculate.

3. **Cell styling**: yellow fill (`FFFFE082`) for input cells, white fill for calculated cells, light gray (`FFF5F5F5`) for headers. Bold for headers, regular for body. Inter font everywhere (or whatever's available in Excel — fall back to Calibri).

4. **Conditional formatting**: ExcelJS supports `cell.conditionalFormatting`. Use it for the green/amber/red severity coloring on margin and discount cells.

5. **Embedded charts**: ExcelJS supports basic charts. The Tab 3 quarterly bar chart and the Tab 8 scatter plot are the main ones. Skip charts on Tab 9 if the implementation gets gnarly — the data tables alone are fine.

### Tier 2: Live Google Sheets integration (Phase 8 — optional)

**Library**: `googleapis` (Google's official Node client).

```bash
npm install googleapis
```

**Setup (one-time, manual)**:

1. Create a Google Cloud project named `kiln-demo`
2. Enable the Google Sheets API and Google Drive API
3. Create a service account, download the JSON key
4. Create a Drive folder `Kiln Demo Workbooks` with public-read permissions
5. Share the folder with the service account's email (gives it write access)
6. Build the master template once: open Google Sheets, replicate the 10-tab structure with the same cell layout as the .xlsx. Save the spreadsheet ID.
7. Add to env vars: `GOOGLE_SERVICE_ACCOUNT_JSON_BASE64`, `GOOGLE_DRIVE_FOLDER_ID`, `GOOGLE_SHEETS_TEMPLATE_ID`

**Server flow** when visitor clicks "Open as live Google Sheets model":

```ts
async function createLiveSheetForDeal(review: DealReview): Promise<string> {
  // 1. Copy the master template
  const copyResponse = await drive.files.copy({
    fileId: process.env.GOOGLE_SHEETS_TEMPLATE_ID,
    requestBody: {
      name: `Kiln — ${review.deal.customer.name} — ${review.deal.name}`,
      parents: [process.env.GOOGLE_DRIVE_FOLDER_ID],
    },
  });
  
  const newSheetId = copyResponse.data.id!;
  
  // 2. Populate cells with the deal's values via batchUpdate
  await sheets.spreadsheets.values.batchUpdate({
    spreadsheetId: newSheetId,
    requestBody: {
      valueInputOption: 'USER_ENTERED',
      data: buildCellUpdates(review),
    },
  });
  
  // 3. Make it readable to anyone with the link
  await drive.permissions.create({
    fileId: newSheetId,
    requestBody: {
      role: 'reader',
      type: 'anyone',
    },
  });
  
  // 4. Return the shareable URL
  return `https://docs.google.com/spreadsheets/d/${newSheetId}/edit?usp=sharing`;
}
```

**UI behavior**:

- Button on the deal review page: "Open as live Google Sheets model"
- On click: shows a brief loading state ("Creating your Sheets model...")
- ~3-5 seconds later: redirects to the Sheets URL in a new tab
- Inline guidance: *"This opens a view-only copy. Use File → Make a copy to get your own editable version."*

**Caching**: cache the created Sheet ID per (deal_id, session_id) pair for 1 hour. If the visitor clicks the button twice, return the same Sheet rather than creating a new one.

**Cost / abuse considerations**: each click creates a Sheet in the Kiln Drive folder. Add a sweep job (or just manually clean up monthly) — the Drive folder shouldn't accumulate thousands of stale Sheets.

## What this signals to the HM

The .xlsx artifact alone signals:

1. **Excel/Sheets fluency** — multi-tab workbooks with proper structure don't happen by accident
2. **Financial modeling** — live formulas with cross-tab references is what real modelers build
3. **ASC 606 fluency** — the rev rec schedule with variable consideration isn't surface-level
4. **Data analysis** — Tab 9's pivot-style aggregations + charts
5. **Production engineering** — programmatic .xlsx generation that supports formulas, formatting, and charts is non-trivial
6. **Deal-desk taste** — the workbook is structured the way an experienced ops person would structure it

The Tier 2 Google Sheets integration adds:

7. **API integration competence** — service accounts, Drive API, programmatic Sheet population
8. **Pragmatism** — choosing the visitor-friendliest path (no OAuth flow) over the "more impressive" but higher-friction one

The combination is a pretty hard package to compete with. Most candidates send a memo. A few send a deck. Almost none send a working financial model that the HM can actually use.
