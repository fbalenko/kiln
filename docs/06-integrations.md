## Slack

**Why it matters**: This is the demo's most visceral "oh, it actually works" moment. The HM watches a deal review in the browser and sees the same review post to a real Slack channel they can join.

### Setup

1. **Create a Slack workspace** named `kiln-demo` (free tier is fine).
2. **Pre-populate channels** so it feels lived-in:
   - `#deal-desk` (primary channel for posts)
   - `#pricing-questions` (a few seeded conversations)
   - `#approvals` (seeded with mock approval threads)
   - `#won-deals` (a few seeded "🎉 closed" celebrations)
   - `#general` (welcome message)
3. **Seed each channel with a week of fake history** — write 30–50 plausible messages from a small set of fake users (4 AEs, 2 RevOps, 1 CFO). The HM should browse channels for 30 seconds and feel like the workspace is real.
4. **Create a Slack app** at api.slack.com:
   - Bot scopes: `chat:write`, `channels:read`, `reactions:read`, `users:read`
   - Install to workspace, copy the bot token
   - Add bot to `#deal-desk` channel
5. **Create a one-click join link** for the workspace and put it in the demo UI. (Workspace settings → "Manage members" → invite link with no expiry.)

**Active invite link for this build**: `https://join.slack.com/t/kiln-demo/shared_invite/zt-3x2f30ipu-EE8uUqOlOCaXR0pruBm0YQ`

This link goes into the "Join the demo Slack" button component on every deal review page. Hardcode it as a constant in `lib/constants.ts` so it's grep-able and easy to update if regenerated.

### Posting contract

When a deal review completes, the orchestrator calls the Slack tool with this payload:

```ts
type DealDeskSlackPost = {
  channel: string;                    // 'C0123456789' (deal-desk)
  blocks: Block[];                    // Slack Block Kit
  thread_ts?: string;                 // for follow-up posts
};
```

Block Kit structure (in order):
1. **Header block** — "🔥 Deal Review: {customer_name} — {deal_name}" (single fire emoji is the only emoji in the entire UI; it's the Kiln brand)
2. **Section block** — deal metadata: ACV, term, type, AE owner
3. **Divider**
4. **Section block with markdown** — the synthesis summary (4 sentences)
5. **Section block with fields** (2-column) — pricing verdict, ASC 606 flag count, redline priority, approval depth
6. **Action block** — three buttons: ✅ Approve, ❌ Reject, 🔍 Needs more info
7. **Context block** — "Generated by Kiln · [view full review](${process.env.NEXT_PUBLIC_APP_URL}/deals/<id>)"

When a button is clicked, an interaction event is sent to `/api/slack-events`. We don't need to actually do anything with it — but log it as an "approval reaction" in the audit log so the HM can see the round-trip works.

### Demo Slack workspace as artifact

- Provide a "Join the demo Slack" button on every deal review page
- Label clearly: "This is a real Slack workspace. The HM can join and watch deal posts arrive in real time."
- After joining, the HM has a persistent breadcrumb in their Slack sidebar — much harder to forget than a closed browser tab.

### Failure mode

If the Slack post fails (network error, rate limit), do NOT block the review. Catch the error, log it, show a "Slack post failed — retry?" button in the artifacts panel.

## Exa (customer signals)

**Why it matters**: Pricing recommendations should be aware of the customer's external context. A customer that just raised $200M is in a different position than one that's laying off staff.

### Setup

1. Sign up at exa.ai
2. Generate an API key
3. Install `exa-js`

### Query strategy

For each deal review, the orchestrator calls Exa with three focused queries:

```ts
async function fetchCustomerSignals(customer: Customer) {
  const queries = [
    `${customer.name} funding round announcement`,
    `${customer.name} executive leadership change`,
    `${customer.name} product launch announcement`,
  ];
  
  const results = await Promise.all(
    queries.map(q => exa.searchAndContents(q, {
      numResults: 3,
      startPublishedDate: subMonths(new Date(), 6).toISOString(),
      useAutoprompt: true,
      type: "auto",
    }))
  );
  
  return mergeAndDedupe(results);
}
```

**Filter by recency**: only include results from the last 6 months. Older context is misleading.

**Cache results** keyed by customer domain + query for 24 hours. Don't burn API credits re-querying the same customer.

### Output rendering

Customer signals appear in the deal detail view as a "Customer Context" card with:
- Headline (truncated to 80 chars)
- Source domain (small, muted)
- Published date
- One-line summary (Exa's auto-generated)
- Relevance score (0–100, computed by the orchestrator's reasoning, not Exa)

Up to 5 signals shown. "View all" expands to up to 9.

### Failure mode

Exa is the most likely external dependency to flake. If it fails:
- The orchestrator emits a `step_complete` for `customer_signals` with an empty results array
- The UI shows "No recent public signals found" — no error banner
- Downstream agents proceed without customer signals (they're context-enhancing, not blocking)

## Vector search (similar past deals)

**Why it matters**: This is the feature that maps most directly to Clay's product DNA. It's a waterfall enrichment for deals — exactly the pattern Clay's customers build inside Clay.

### Implementation

Use `sqlite-vec` extension on the same SQLite DB. No separate vector store needed.

```ts
// Embed text using OpenAI (or Anthropic embeddings if available)
// text-embedding-3-small: 1536 dimensions, $0.02 / 1M tokens
async function embedDeal(deal: Deal): Promise<Float32Array> {
  const text = `
Customer: ${deal.customer.name} (${deal.customer.segment}, ${deal.customer.industry})
Deal type: ${deal.deal_type}
ACV: $${deal.acv}, term: ${deal.term_months}mo
Discount: ${deal.discount_pct}% — ${deal.discount_reason}
Non-standard clauses: ${deal.non_standard_clauses?.join(", ")}
Customer request: ${deal.customer_request}
Competitive context: ${deal.competitive_context}
  `.trim();
  
  const resp = await openai.embeddings.create({
    model: "text-embedding-3-small",
    input: text,
  });
  
  return new Float32Array(resp.data[0].embedding);
}
```

### Search query

```sql
SELECT 
  deal_id,
  distance
FROM deal_embeddings
WHERE embedding MATCH ?
  AND k = 4              -- top 4, exclude self
ORDER BY distance ASC
LIMIT 4;
```

Filter out the source deal itself. Return top 3 of the remaining.

### Output rendering

Each similar deal renders as a card in the "Similar past deals" panel:
- Customer name + deal headline
- Outcome badge (won / lost / closed)
- A one-line "decision note" — pulled from the deal's review record (e.g., "Accepted MFN with 12-month carve-out + price floor protection")
- Distance score (shown as "85% similar" rather than raw cosine)
- Click to navigate to that deal's full review

### Visitor-submitted deals

When a visitor submits their own deal:
1. Compute embedding at submit time
2. Run k-NN against the seeded set
3. Return top-3 similar deals
4. Do NOT persist the embedding — it's transient

This way the visitor sees their own deal placed in context against our seeded institutional memory.

## Document generation

**Why it matters**: The HM should leave the demo with a folder of generated artifacts. Tangible deliverables transform the artifact from "demo" to "tool."

### Five artifacts per review

1. **Redlined MSA (.docx)** — A Master Services Agreement template with the customer-proposed clauses inserted, our suggested counter-positions shown as tracked changes. Generated using the `docx` package.

2. **Populated order form (.pdf)** — A clean PDF with deal metadata, pricing breakdown, ramp schedule, ASC 606 treatment notes. Generated using `pdfkit`.

3. **AE email draft (.eml or markdown)** — Email body to the AE with subject line, structured to-do list, and counter-proposal language. Renders inline in the UI; downloadable as `.eml` for opening in mail clients.

4. **Approval review one-pager (.pdf)** — Single-page PDF for the deal-desk review meeting: deal summary, key recommendations, approval routing, the orchestrator's synthesis. Generated using `pdfkit`.

5. **Financial model workbook (.xlsx)** — A 10-tab workbook with live formulas covering deal summary, pricing model, ASC 606 revenue recognition schedule, alternative structures, approval routing, approval matrix (editable), pricing guardrails, similar deals, comp analysis, and audit log. Generated using `exceljs`. **See `docs/10-sheets-integration.md` for full spec** — this is the artifact that directly addresses the JD's "Advanced Excel/Google Sheets skills, financial modeling, ASC 606" requirements, plus an optional Phase 8 Google Sheets API integration that opens the workbook live in Sheets.

### Generation strategy

Don't pre-generate. Generate on-demand when the visitor clicks the download button. Cache the result for 5 minutes (in case they click twice). This keeps the demo's cold-start latency low.

### Templates

Templates live in `lib/document-templates/` as TypeScript files (not template engines — direct programmatic generation). Each takes the `DealReview` object as input and produces a Buffer.

### Visual design

- Headers in the same Inter typeface
- Margin: 1 inch all sides
- Footer: "Generated by Kiln · [${process.env.NEXT_PUBLIC_APP_URL}/deals/<id>]"
- The first page of every PDF includes the disclaimer: *"This document was generated by an AI system as a demonstration. Final approval requires human review and applicable jurisdictional legal review."*

### Failure mode

Generation failures are rare but possible. If a doc fails to generate, the download button disables and shows a tooltip: "Generation failed. Try again in a moment."

## MCP server architecture

All integrations are exposed as MCP servers running in-process within the Next.js app. This keeps the architecture clean and allows the prompts/tools to be forked by Clay if they wanted to.

```
lib/mcp-servers/
├── crm-server.ts            # CRM read tools
├── pricing-server.ts        # Guardrails, list price lookups
├── exa-server.ts            # Customer signals
├── vector-server.ts         # Similar deals
├── slack-server.ts          # Posting + reactions
├── document-server.ts       # PDF/DOCX/email generation
└── audit-server.ts          # Audit log writes
```

Each server registers tools using the MCP SDK. The orchestrator and sub-agents declare which tools they have access to via the Claude Agent SDK's tool restriction config.

### Tool naming convention

`<server>.<verb>_<noun>` — e.g., `crm.get_deal`, `exa.customer_signals`, `slack.post_deal_review`. This pattern shows up in the audit log and makes it grep-able.
